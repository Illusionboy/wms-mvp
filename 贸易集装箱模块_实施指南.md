# 贸易集装箱模块 — 实施指导文件

> 本文件是对 `贸易集装箱模块.md`（原始设想）的评估结论 + 面向 CLAUDE 的实现规范。
> 后续实现须严格遵循本文件；与 `CLAUDE.md` 的项目铁律冲突时以 `CLAUDE.md` 为准。

## 背景与目标

以**托盘（pallet）**为核心的集装箱备货/装柜模块，把"客户预留（应发什么）"落到"托盘/装柜（实际装了什么）"，经多道人工核对后一键出库。要求：与现有流程零冲突、**可分阶段灰度**（出库扣库存先不启用）。

深度依赖已有**客户预留模块 `CustomerAllocation`**。

**关键业务背景（WMS 非实时在库）**：到货后先在**秦丝生意通报库**，当天或次日盘点检查后**才同步进 WMS**；但报库后**第一时间就分货装托盘**。因此装托盘时 WMS 往往还没同步这批入库，货虽**实物已到**、预留却显示 `waiting`。由此推出三条设计基准：① 本模块把 `reserved` + `waiting` **都当作可装载**；② 对账以"预留需求量"为基准，**不以 WMS 在库为准**（此阶段 WMS 在库不可信）；③ 这也是"集装箱出库扣库存必须先灰度关闭"（风险④）的额外硬理由——装托盘时 WMS 入库尚未到位，此刻扣减会打成负库存。

### 必须遵守/复用的既有事实

- `CustomerAllocation`：唯一键 `(planned_outbound_date, customer_name, jan_code)`，字段 `quantity/status`。状态：
  - `waiting`（WMS 记录库存不足；**但本项目因 WMS 滞后于实物，`waiting` 常 = 货已到、WMS 未同步 → 属可装载状态**，见上方"关键业务背景"）
  - `reserved`（已"调转"=软预留，**不扣减** `InventoryRecord`）
  - `shipped`（已实际出库，**仅状态标记**，扣库存发生在别处）
  - `cancelled`
- 预留只针对**普通仓库**（`ALLOCATION_WAREHOUSE_NAME`）。`customer_name` 是客户代码字符串（mm/kk/cp/hn/xm/mmm…）。
- 复用（不要重写）：
  - `app/services/customer_allocations.py`：`get_daily_allocation_overview(date)`、`get_allocation_status(...)`、`get_allocation_summary(customer,date)`、`mark_as_shipped(allocation_id)`
  - `app/services/inventory.py`：`stock_out_item(...)`（写 `StockTransaction`、带 `customer`、触发 `check_for_reservation_conflict`）、`resolve_canonical_jan(...)`（别名归并）
  - 前端扫码模块（`app/static/app.html`）：`openScanner`、`scanIntoRawJan`、`scanIntoWidget`
- 铁律：每笔库存变动必写不可变 `StockTransaction`（带 `user_id`、`source`、幂等 `reference_id`）；记录不删；批量走 Draft-Apply + `SELECT FOR UPDATE` 行锁；歧义不猜；新表在 `init_db.py` 建表；变动端点加 `require_auth`。**新增为主，绝不改动共享的出库/预留逻辑。**

## 一、评估结论

**合理、保留**：托盘颗粒度对齐预留（date+客户+JAN+量）、"只能从预留选不能手输"、多道人工 CHECK、复用扫码——方向正确。

**已识别风险 + 处理（均经用户确认）**

| # | 风险 | 处理方案 |
|---|---|---|
| ① | `waiting` 语义：一般 WMS 里=库存不足；但本项目 WMS 滞后实物，`waiting` 常="货已到、WMS未同步" | **加货候选 = 该客户该日期 `reserved` + `waiting` 并集，两者都可装**（用户确认）；`waiting` 行只附**中性信息标签**（如"WMS未同步/在库 N"），**不拦截、不用红字警告**；对账基准用预留(需求)总量，不用 WMS 在库 |
| ② | 一 JAN 的量可能跨多个托盘，单托盘比对会漏算/重算 | **对账一律按 (客户, 日期, JAN) 跨该客户当天所有托盘求和** 后再与预留总量比较 |
| ③ | 第7点"装柜后清空托盘"与第8点"卦柜前复核漏装"自相矛盾 | **托盘内容（`PalletItem`）保留到最终"一键出库"完成后才清空**；装柜时同步写入不可变快照 |
| ④ | 预留不扣库存 → 其他渠道（秦丝/贸易出库/web）可能重复发/数同批货 | **集装箱出库代码写好但配置开关默认关**：试运行期实际扣库存仍走贸易出库，两套并存不干扰；流程跑通形成 SOP 后再开开关切换 |
| ⑤ | 托盘身份与一次装柜内容混淆 | 拆三层：`Pallet`（永久 ID）/ `PalletItem`（当前内容）/ `ContainerLoadItem`（装柜快照） |
| ⑥ | NFC = Web NFC，**仅安卓 Chrome + HTTPS**，iPhone Safari 完全不支持 | 托盘身份**主方案 = 条码**（复用摄像头扫码，跨平台）；NFC 作安卓端增强（特性检测，iOS 隐藏）；蓝牙打印条码属 P3 |
| ⑦ | 柜 ↔ 客户/日期基数未定 | **一天可多客户混装一柜**，需独立 `Container` 实体 |
| ⑧ | 并发 | 托盘加货/合并/转移/出库都加行锁；两托盘合并/转移按 `min(pallet_id)` 规范锁序防死锁 |
| ⑨ | 跨托盘合并/转移会破坏对账口径 | 合并/转移**仅限同客户 + 同日期**的托盘之间；移货只改 `PalletItem`、不动库存 |
| ⑩ | 客户名格式不一致导致对不上账 | 托盘 `customer_name` 必须从当天预留客户列表里**选**（禁止自由文本），与 `CustomerAllocation.customer_name` 同值同格式 |

## 二、数据模型

新表放 `app/models/`，均继承 `Base, TimestampMixin`；在 `app/models/__init__.py` 注册；在 `app/db/init_db.py` 建表。

- **`Pallet`**（物理托盘，永久存在）
  - `id`、`code`（贴在托盘上的条码/NFC 值，**unique**）、`shelf_location`（可空，**沿用项目现有库位规范 `区-排-层`，如 `A-12-03`**，`String(32)`；托盘身份码与库位码是两码，见下方"库位绑定"）、`customer_name`（可空，当前绑定）、`planned_outbound_date`（可空，当前绑定）、`status`（`empty` / `staging` 备货中 / `loaded` 已装柜）、`note`
- **`PalletItem`**（托盘当前内容，可变）
  - `id`、`pallet_id` FK、`jan_code`、`quantity`；唯一约束 `(pallet_id, jan_code)`
- **`Container`**（装柜批次，支持一柜多客户）
  - `id`、`container_no`（可空）、`seal_no`（卦签/封签，可空）、`outbound_date`、`status`（`loading` / `sealed` / `shipped`）、`note`
- **`ContainerLoadItem`**（装柜快照，**不可变**，写入后不改）
  - `id`、`container_id` FK、`pallet_id` FK、`jan_code`、`quantity`、`customer_name`、`planned_outbound_date`、`loaded_at`、`user_id`
  - 第8点卦柜前复核、第9点一键出库都以本表为准

**对账恒等式**：应装（预留） vs 实装（`PalletItem` 或 `ContainerLoadItem`），一律按 `(customer_name, planned_outbound_date, jan_code)` 聚合。

## 三、后端

先在 `app/schemas/` 写 DTO；业务逻辑放 `app/services/trade_containers.py`；端点放 `app/api/v1/endpoints/trade_containers.py` 并在 `app/api/v1/router.py` 注册。查询端点只读免认证，变动端点加 `require_auth`，`source="trade_container"`。

**服务函数要点**

- `create_empty_pallet(code, date, customer)`：校验该托盘为空（无 `PalletItem`）；`date`/`customer` 必填；`customer` 必须来自 `get_daily_allocation_overview(date)` 返回的客户列表（只能选，不能手输）。
- `place_pallet_at_location(pallet_code, location_code)`（库位绑定/放置）：扫托盘码 + 扫库位码后调用；校验 `location_code` 符合 `区-排-层` 格式（如 `A-12-03`）→ 更新对应 `Pallet.shelf_location`。**仅改字段、不动库存、幂等**（重复放同一库位无副作用）。库位标签建议 **1D Code128**（贴货架横梁、横向扫更顺手；QR 亦可但 1D 更贴合货架边条）。托盘身份码用 QR（见风险⑥），两者是**两个独立码**。
- `add_items_to_pallet(pallet_id, [(jan, qty)])`：候选 = 该客户该日期 `reserved` + `waiting` **并集**（两者都可装；`waiting` 行附中性信息标签"WMS未同步/在库 N"，不拦截，因 WMS 滞后实物，见"关键业务背景"）；数量**手动**输入；不在预留中的 JAN 可手动加（扫商品码/外箱码，用 `resolve_canonical_jan` 归并别名）；**绝不修改预留**。行锁托盘行。
- `merge_pallets(src, dst)` / `transfer_items(src, dst, [(jan, qty)])`：仅同客户 + 同日期；`min(pallet_id)` 锁序；只改 `PalletItem`，不动库存；转移只能点选源托盘已有的 JAN。
- `preday_reconcile(date)`（第6点，出库前一天核对）：应装 vs 当前 `PalletItem` 求和，按客户列出差异；附带"某客户有几个托盘"。
- `verify_pallet_against_reservation(container_id, pallet_id)`（第7点，装柜扫描核对）：该托盘 + 本柜已装 `ContainerLoadItem` 聚合 vs 预留 → 返回 `一致` / `多货` / `不在预订`。
- `load_pallet_to_container(container_id, pallet_id)`：一致时写 `ContainerLoadItem` 快照 + 托盘 `status=loaded`（**不清 `PalletItem`**）。
- `preseal_reconcile(container_id)`（第8点，卦柜前漏装核对）：应装（该柜涉及日期）vs `ContainerLoadItem`(container) 聚合；漏装的列出托盘 ID + `shelf_location`；预留改单后**重跑本函数即可复核**，无需重扫托盘。
- `ship_container(container_id)`（第9点，一键出库；受配置开关 `settings.trade_container_stockout_enabled` 控制，**默认 False**）：
  - **False（试运行）**：只写装柜/出库事件 + 清空各托盘 `PalletItem`（托盘回 `empty`）；**不扣库存、不改预留状态**（实际出库仍由贸易出库负责）。
  - **True（SOP 后）**：对每条 `ContainerLoadItem` 调 `stock_out_item`（`source="trade_container"`，幂等 `reference_id="trade_container:{container_id}:{pallet_id}:{jan}"`）+ `mark_as_shipped` 对应预留；再清空托盘。

**配置**：在 `app/core/config.py` 新增 `trade_container_stockout_enabled: bool = False`（env `TRADE_CONTAINER_STOCKOUT_ENABLED`）。

## 四、前端（`app/static/app.html`）

### 4.1 信息架构（IA）：集装箱是独立「工作区」，不塞进日常导航

**问题**：主 SPA 侧边栏已有 5 组约 18 项；集装箱模块自身 7~8 个子屏，且属**装卸货现场、不同人、不同场景**的作业，直接并入现有导航会把主页堆乱。

**原则**（信息架构通用做法）：导航按**使用场景/角色**切成「工作区（workspace）」，而非把所有功能平铺在一个侧边栏——一次只暴露当前场景要用的入口，降认知负荷、可持续扩展。

**本项目落地规则（照此实现）**：

1. **仍在同一个 `app.html` 内实现，不新开 HTML 页**。理由：直接复用已有 `api()` / 登录态 / `openScanner`（含已内置 ZXing）/ `toast` / `nav()`，**零复制粘贴**。另开独立页（如 legacy `count_import.html`）会被迫复制这些共享逻辑、重复引入 ~300KB 扫码库，且与"往 SPA 收敛"的方向相悖——**明确不采用**。
2. 侧边栏顶部加**工作区切换**（分段控件/下拉）：`仓库作业`（默认，现有 5 组原样）｜`集装箱装柜`。给每个 `nav-group` 打 `data-workspace="ops|container"`；`switchWorkspace(name)` 只 show/hide 对应分组，并同步 `location.hash`（如 `#ws=container`）+ `localStorage` 记住上次工作区（便于深链/收藏、现场直接进）。
3. `集装箱装柜`工作区有**自己聚焦的导航**（见 4.3），且**移动端/扫码优先**布局（大按钮、少层级——现场单手扫）。
4. 以后新的大模块同样各成一个工作区，侧边栏永不堆成一面墙。

> 备选（不推荐）：独立页 `/container`→`container.html`，隔离最彻底、可单独收藏；但需复制 auth/api/scan 三套共享逻辑、易漂移，与本项目"SPA 收敛、每页自包含无构建"现状冲突。除非坚持完全隔离，否则用上面的工作区方案。

### 4.2 复用与交互

- 托盘身份：主用摄像头扫码（复用 `openScanner`）；安卓额外支持 Web NFC（`NDEFReader`，特性检测，iOS 隐藏按钮）。
- 加货界面复用 `scanIntoWidget` / `scanIntoRawJan`；候选列表 = 该客户该日期 `reserved` + `waiting` 并集（`waiting` 行显示中性状态芯片如"待WMS同步"，不用红字警告、不禁选）。

### 4.3 `集装箱装柜`工作区的子屏

section 仍写作 `<div class="section" id="section-container-*">`，复用现有 `nav(this)` 切换；工作区切换只控制哪批 nav-group 可见（这些 section 默认随工作区隐藏）。子屏：

托盘查询 / 建空托盘 / 加货 / **库位绑定（扫托盘码 + 扫库位码 → 确定放置；两扫码框各复用 `openScanner`，确定后调 `place_pallet_at_location`）** / 出库前对账（含"某客户几个托盘"）/ 装柜扫描核对（一致·多货·不在预订三分支）/ 卦柜前漏装复核 / 一键出库（试运行期按钮明确标注"不扣库存，仅记录"）。

## 五、验证方式

造数据：普通仓库若干 JAN 库存 + 上传某日期多客户预留（得到 reserved/waiting）→ 建空托盘 → 加货（验候选=reserved+waiting 并集、waiting 可选不禁用、显示中性状态芯片）→ 出库前对账 → 建柜 → 装柜扫描核对（造一致/多货/不在预订三种）→ 卦柜前漏装复核（改预留后重跑）→ 一键出库：

- 开关 **False**：确认**无** `StockTransaction`、预留状态不变、托盘清空、快照留存、贸易出库照常可用。
- 开关 **True**：确认写 `StockTransaction`（OUT，source=trade_container）、预留变 `shipped`、重复提交幂等跳过。

回归：确认既有预留 / 贸易出库 / 秦丝流程零改动。

## 六、分期实施

- **P1**：数据模型 + 建托盘 / 加货 / 库位绑定 / 托盘查询 + 前端骨架（**工作区切换 + `集装箱装柜`工作区**，见四.1）+ 扫码接入。
- **P2**：对账（出库前 / 装柜核对 / 卦柜复核，均聚合）+ 建柜 / 装柜快照 + 一键出库（带开关，默认关）。
- **P3**：托盘合并 / 转移、Web NFC、蓝牙打印条码标签、开关切 True 的 SOP 校验。
