## 1. 一句话概述（TL;DR）

一套**自托管（NAS / Docker）的仓库库存管理系统（WMS）**，核心价值不是"又一个进销存"，而是：

> **把分散在 ERP（秦丝生意通）、电商平台（乐天 RMS）、即时通讯（微信报库）三条互不相通的数据流，统一收敛进一张不可变的事务事实表（`stock_transactions`），并在其上构建时序分析、历史追溯与统计学库存预警能力。**

技术栈：FastAPI + PostgreSQL 16（async SQLAlchemy 2.0）+ Gemini（非结构化文本解析）+ aiogram（Telegram 只读查询）。前端是一个零依赖、可离线运行的单页应用（`app.html`）。

## 2. 缘起与构思（"为什么做"）

作者在实际运营一个跨境/批发仓库，每天面对的真实痛点是：

1. **数据源彼此割裂**：
   - 采购入库 / 国内销售出库记录在**秦丝生意通（ERP）**里；
   - 乐天（Rakuten）店铺的 B2C 出货是**平台导出的 CSV**；
   - 临时的进出货靠**微信群里发文字消息**报库；
   - 没有任何一个地方能看到"这个商品现在到底有多少、最近怎么动的"。

2. **盘点严重滞后**：公司每月底盘点，但盘点统计结果可能**到次月 5 号才出来**。这段窗口期里出入库一直在发生 —— 如果系统强制库存不能为负，操作员根本没法录单。

3. **商品编码混乱**：乐天订单里的"商品番号 + システム連携用SKU番号"要经过一套**非平凡的规则**才能还原成真正的 13 位 JAN 码（尤其是套装商品 / `-0` 可变套装），人工对照极易出错。

4. **没有历史、没有分析**：老板想问"这个客户上个月拿了多少货""哪个供应商供货最频繁""哪些 SKU 该补货了"，没有任何系统能回答。

> **构思的核心判断**：问题的本质不是"缺一个记库存的工具"，而是"**缺一个能把多源数据标准化、留痕、可追溯、可分析的数据底座**"。所以从第一天起，系统就是围绕"**事务即事实（每一次库存变动都是一条不可删改的记录）**"来设计的，而不是围绕"一个会被反复覆盖的库存数字"。

## 3. 关键设计决策与思考过程（"怎么想的" —— 本文档最有价值的部分）

下面每一条都是一个**有意识的架构取舍**，面试中最能体现"工程/数据思维"的正是这些"为什么"。

### 3.1 事务是唯一事实来源（Event-sourcing 式设计）
- **决策**：任何库存变动都**必须**写一条 `StockTransaction`（IN / OUT / ADJUST），**绝不允许**只改 `InventoryRecord.quantity` 这个汇总数字。记录**永不修改、永不删除**（所有外键 `ON DELETE RESTRICT`）。
- **为什么**：库存数字只是"当前快照"，会被覆盖；而事务流是"历史真相"。只要事务流完整，任何时点的库存、任何维度的分析都能重算出来。这等价于会计的"流水账 vs 余额"，也是数据仓库里**事实表**的思路。
- **专业意义**：这是"可审计、可追溯、可分析"的前提，直接对应数据治理与数仓建模意识。

### 3.2 `transaction_date` 与 `created_at` 分离（时序分析的主轴）
- **决策**：每条事务有两个时间 —— `transaction_date`（**实际业务发生日期**）和 `created_at`（**系统录入时间**）。两者分开存。
- **为什么**：秦丝订单可能 3 天前发生、今天才同步进来；盘点调整发生在盘点当天、却在次月录入。如果只用录入时间做趋势分析，时间轴是错的。**业务日期才是需求预测、周转率、季节性分析的正确横轴**。
- **现状（重要）**：秦丝同步、盘点 ADJUST、手动录入**已写入** `transaction_date`；**乐天 CSV / 微信报库 / 贸易出库目前仍为 NULL**（查询时回退到 `created_at`），这是已知待补的数据质量缺口（见第 9 节）。

### 3.3 多源整合靠统一的 `source` 字段
- **决策**：所有事务汇入同一张表，用 `source` 区分来源：`qinsi_scrape`（秦丝 ERP）/ `rakuten_csv`（乐天 TMS 类）/ `chat_report`（微信 AI 解析）/ `physical_count`（盘点）/ `web_ui`（手动）/ `trade_shipment`（贸易批发）/ `rakuten_order`。
- **为什么**：一张统一事实表 + 一个渠道维度，就能做**跨渠道对比**（B2C 乐天 vs B2B 批发 vs 国内 ERP），而不需要在多个系统间手工拼数据。这正是物流分析岗位要求的"整合 ERP+WMS+TMS 多系统数据"。

### 3.4 Draft-Apply 两步流（批量操作的安全闸）
- **决策**：所有批量导入（聊天记录、乐天 CSV、盘点表、贸易出库、乐天采购）都走"**先生成草稿 → 人工预览确认 → 再 Apply 写库**"两步，Apply 时用 `SELECT FOR UPDATE` 锁住草稿行防重复提交。
- **为什么**：AI 解析和文件导入都可能出错，**批量误操作的代价很高**。两步流把"机器解析"和"人工确认"解耦，是数据质量管控的关键设计。

### 3.5 幂等（Idempotency）`reference_id`
- **决策**：可能重复导入的来源都生成确定性的幂等键，如秦丝 `qinsi:{order_no}:{jan_code}`、贸易出库 `trade:{draft_id}:{line_index}`，在 `(source, reference_id, transaction_type)` 上加唯一约束，重复提交自动跳过。
- **为什么**：网络重试、用户重复点击、同一份文件传两次，都不能造成重复扣库存。幂等是数据可靠性的底线。

### 3.6 冷启动期的"负库存"策略（对业务节奏的理解）
- **决策**：每个仓库有 `allow_negative_stock` 开关。开启后，库存不足时**不报错、直接扣成负数并留痕**；DB 层也**故意不加** `CHECK(quantity >= 0)` 约束。盘点数据到位后，再用 `physical_count` 的 ADJUST 事务把余额修正到实际值。
- **为什么**：这是对前述"盘点滞后窗口期"业务现实的直接回应。**与其让系统教条地阻止操作员录单，不如允许负库存、留下痕迹、事后对账修正**。仪表盘上"负库存 SKU"可点开查看明细，负值本身就是"待对账"的信号。
- **专业意义**：体现对 3PL/仓储真实操作节奏的理解 —— 系统要服务业务，而不是反过来。

### 3.7 盘点对账公式（对 WMS 业务逻辑的深度理解）
- **决策**：盘点不是简单地"用盘点数覆盖系统库存"，而是：
  ```
  目标库存 = 盘点数量(盘点日) + Σ(盘点日之后的所有 WMS 出入库变动)
  ADJUST量 = 目标库存 − 当前 WMS 库存
  ```
- **为什么**：盘点是过去某一天的物理快照，而录入是在之后才完成的；这期间又有新的出入库。**简单覆盖会抹掉窗口期内的真实变动**。这条公式是整个盘点工作流的灵魂，也是面试里能体现"懂业务、不只是写代码"的硬核细节。

### 3.8 库存桶（Bucket）模型的刻意简化
- **决策**：`InventoryRecord` 实质上只按 **`(product_jan, warehouse_id)`** 一桶一记录。客户、库位、效期等维度在 DB 约束里保留但实际恒为 NULL/默认值；**客户绝不参与库存桶定位**。
- **为什么**：MVP 阶段过早引入库位/批次/客户多维度会让库存逻辑爆炸复杂。先把"商品×仓库"这条主线做扎实，把客户/供应商信息放到**事务的结构化列**里供分析用（见 3.9），而不是塞进库存桶。这是"**对的抽象层级**"的取舍。
- **预留扩展**：文档里已规划 FEFO（先到期先出）只需把选桶排序从 `id ASC` 改成 `expiration_date ASC NULLS LAST`，是单行改动 —— 说明简化是"留了口子的简化"。

### 3.9 `customer` / `supplier` 升级为结构化列
- **决策**：交易对手方不再只塞进自由文本 `note`，而是 `StockTransaction.customer`（出库对方）/ `supplier`（入库对方）两个**可查询的结构化列**。
- **为什么**：放在 `note` 里只能给人看、没法聚合查询。结构化后，"按客户查出库历史""按供应商查入库历史""所有交易对手列表"这些分析端点才成立。**这是从"记录"到"分析"的关键一跃**。

### 3.10 共享 JAN 解析纯函数（复杂业务规则的收敛）
- **决策**：把"乐天订单行 → JAN+数量"的复杂规则抽成一个**纯函数** `resolve_jan_quantities()`，配一份兜底字典 `product_dict.json`，被"乐天出库""乐天采购分析"两处复用；规则细节单独写成 `JAN_QUANTITY_SPEC.md` 规格文档。
- **为什么**：套装商品、`-0` 可变套装、多零后缀等规则极易出错且会变。**纯函数 + 规格文档 + 单一事实来源**，避免同一套规则在多处各写一遍、改了一处漏了另一处。（注：该函数在快递工具子项目里有一份副本，规则变更需两边同步。）

## 4. 系统架构总览

### 4.1 技术栈
| 层 | 选型 | 说明 |
| --- | --- | --- |
| 后端 | FastAPI (Python 3.11+) | 全异步 |
| 数据库 | PostgreSQL 16 + asyncpg | SQLAlchemy 2.0 async，**禁止裸 SQL / 同步会话** |
| 认证 | JWT (HS256, 30天) + bcrypt | 所有**写**接口需认证；**读**接口（查询/分析）开放 |
| AI 解析 | Google Gemini (`gemini-2.5-flash`) | 把非结构化文本/图片解析成结构化 JSON；prompt 用 `<chat_log>` 标签包裹防注入 |
| ERP 客户端 | httpx（无浏览器） | 直接调秦丝 API；登录需滑块 CAPTCHA，cookie 7 天有效，本地 Mac 续期后上传 NAS |
| IM | aiogram 3 (polling) | Telegram bot **只读查询**，不做任何写操作 |
| 文件 | openpyxl(Excel) / BeautifulSoup(HTML) / CSV | 商品目录、盘点表、乐天 CSV、秦丝盘点 HTML |
| 部署 | Docker Compose | postgres / api / bot 三服务，NAS 自托管 |

### 4.2 核心数据流
```
       ┌─ 秦丝ERP (httpx API)  ──┐
       ├─ 乐天RMS CSV          ──┤   source 字段区分渠道
       ├─ 微信文字 (Gemini)    ──┼──►  stock_transactions  ──►  inventory_records (当前快照)
       ├─ 贸易出库 Excel/拍照  ──┤     (不可变事实表)              │
       ├─ 月度盘点 HTML/Excel  ──┤                                 ├──►  analytics/* 历史查询
       └─ 手动 Web UI          ──┘                                 └──►  安全库存预警 / KPI(规划)
```

### 4.3 数据模型主干
```
stock_transactions  ──►  inventory_records  ──►  products
   (事实表)                (商品×仓库快照)        (JAN为主键)
       │                        │
       │                        └──►  warehouses (含 allow_negative_stock)
       └── 关键分析字段：transaction_date / source / customer / supplier /
                          transaction_type / quantity_change / reference_id
```

## 6. 工程化与质量控制亮点（面试可讲）

- **并发安全**：同桶并发出库 `SELECT FOR UPDATE`；入库防重复建桶行锁；草稿防重复提交行锁；低库存标志位防竞态翻转。（已识别多用户阶段的死锁/重投/连接池/盘点顾问锁方案，存档待实现。）
- **幂等网**：唯一约束 `(source, reference_id, transaction_type)`，多源导入安全可重试。
- **Schema 演进**：启动时 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` 自动迁移（适合自托管、无专职 DBA 的场景）。
- **离线优先**：前端零 CDN 依赖、纯内联 JS/CSS，NAS 断网可用。
- **Prompt 安全**：Gemini 输入用标签包裹 + 防注入指令。
- **规格驱动**：复杂的 JAN 解析规则单独写规格文档（`JAN_QUANTITY_SPEC.md`），纯函数实现、多处复用。

## 7. 从供应链 / 物流分析的专业视角解读（求职定位的核心）

> 这一节是把"一个仓库工具"翻译成"**物流数据分析师的能力证据**"。对标岗位：**Kering Japan – Logistics Business Analyst**（汇报给 Logistics Cost Control & Projects Manager）。完整逐条差距分析见仓库内 `docs/kering_interview_gap_analysis.md`。

### 7.1 已能正面陈述的能力（已完成，可直接讲）
1. **多系统数据整合**：ERP（秦丝）+ 电商 TMS（乐天）+ IM AI 解析 → 统一事实表 + `source` 渠道维度 → 跨渠道分析。**直接命中 JD "整合并分析多系统的出货与成本数据"。**
2. **数仓建模意识**：`transaction_date`（业务日期）与 `created_at`（录入时间）分离，是一切趋势/季节性/周转分析的正确时间轴 —— 体现对事实表/维度建模的理解。
3. **WMS 业务深度**：盘点对账公式（不是简单覆盖）、负库存冷启动策略 —— 体现懂仓储/3PL 真实操作节奏，不是纸上谈兵。
4. **数据质量管控**：Draft-Apply 两步流、幂等 `reference_id`、行级锁 —— 是"数据可信"的工程保证。
5. **统计学库存模型**：动态安全库存 `SS=Z·σ·√L` + 再订货点 ROP，按供应商交期分档 —— 体现把统计方法落到供应链决策上的能力。

### 7.2 专业框架映射（把项目术语翻成行业语言）

### 7.3 已识别的差距（诚实披露 —— 这些是"规划/待补"，**勿当已完成对外说**）
对标 Kering JD 仍缺：
1. **无成本/费用数据**：没有 `unit_cost`、没有物流费用表 → 暂时**做不了 invoice validation / cost per unit shipped 分析**。（JD 重点要求，差距最大。）
2. **无 BI 工具对接**：报表都在 Web UI 内，**还没有 Power BI / Tableau 连 PostgreSQL 的成品**。JD 明确要 development-level Power BI。
3. **无品牌/渠道（B2B vs B2C）维度的成型报表**：字段层面有 `source`，但还没做成"每品牌周报"那种交付物。
4. **无量化 KPI 视图**：库存周转率、损耗率、订单满足率等还停在 SQL 草稿（`docs` 里有视图示例），**未落到仪表盘**。
5. **`transaction_date` 未补全**：乐天 CSV / 微信 / 贸易出库仍为 NULL，时序分析有空洞。
6. **无自动化报告分发**：没有定时生成 Excel/邮件/Telegram 日报的能力。
