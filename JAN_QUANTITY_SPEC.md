# 乐天订单 JAN 数量解析规范

本文档描述如何从乐天 RMS 导出的订单 CSV 中准确解析每行订单的 JAN Code 和实际发货数量。
该逻辑在乐天快递文件生成项目（`merge_tool.py` → `merge_product_columns`）中已实现，
本文档供库存统计、乐天库存更新等下游系统直接复用。

---

## 一、数据来源

乐天 RMS 后台 → 注文管理 → 注文详細CSV（第二种，最详细版）。
导出文件为 cp932（Shift-JIS）编码。

**必须读取的列**（强制声明为 `str` dtype，防止 pandas 截断前导零或长整数）：

| 列名 | 用途 |
| --- | --- |
| `商品番号` | 商品主键，格式见下节 |
| `システム連携用SKU番号` | 变体/套装 JAN，权威来源 |
| `個数` | 客户订购件数 |

---

## 二、`システム連携用SKU番号` 是权威来源

管理员已在 RMS 后台将每个 SKU 的 `システム連携用SKU番号` 填写为标准格式。
**实际解析时以此列为主，`商品番号` 仅作为前置过滤用。**

### 格式 1：普通单品

```
JAN13位[-套数]
```

| 例子 | JAN | 套数 | 说明 |
| --- | --- | --- | --- |
| `4971710376227-1` | 4971710376227 | 1 | 单件 |
| `4971710376227-2` | 4971710376227 | 2 | 两件装（每订购1套出2件） |
| `4530025011829` | 4530025011829 | 1 | 无后缀，套数=1 |

**套数后缀解析规则**：对 `システム連携用SKU番号` 从右切分一次（`rsplit('-', 1)`），
当且仅当右侧部分是 **1~3 位纯数字** 时，才视为套数；否则整体视为 JAN。

```
套数 = 0  →  强制视为 1
```

**最终发货数量** = 套数 × `個数`

---

### 格式 2：套装商品（多款合一）

当一次订单包含多款不同 JAN 时，格式为：

```
完整JAN(13位) - 4位局部代码 [- 4位局部代码 ...] - 件数
```

**示例**：`4971710376227-7002-2`

| 字段 | 值 | 含义 |
| --- | --- | --- |
| 第一部分 | `4971710376227` | 第一款商品的完整 JAN |
| 中间部分 | `7002` | 第二款商品 JAN 的后 4 位 |
| 最后部分 | `2` | 套装内共 2 款商品 |

**后续 JAN 重建规则**：取第一个 JAN 的**前 9 位**作为公共前缀，拼接 4 位局部代码：

```
第一个 JAN:  4971710376227
前 9 位:     497171037
第二款 JAN:  497171037 + 7002 = 4971710377002  ✓
```

**套装检测条件**（同时满足）：
1. 按 `-` 分割后有 ≥ 3 段
2. 第一段是 13 位纯数字（完整 JAN）
3. 中间各段均为 4 位纯数字
4. 最后一段是纯数字（件数）

**每款商品的发货数量** = 1 × `個数`（套装中每款各一件，件数字段只表示包含的款数）

## 三、完整解析流程

```
输入：商品番号, システム連携用SKU番号, 個数

1. 解析 商品番号
   → 按 1~3 位纯数字后缀规则拆出 (barcode, set_count)
   → 若后缀为 0：标记为「可变套装/选项商品」(is_variable_set)，
     不使用 set_count，直接进入 Step 2 —— 该商品有多种套装/选项，
     实际 JAN 和数量必须以 システム連携用SKU番号 为准
   → 若 barcode 为 13 位纯数字 且 非可变套装：直接使用，输出 {barcode: set_count × 個数}
   → 否则：进入 Step 2

2. 解析 システム連携用SKU番号
   a. 套装检测（见上节）→ 输出多个 {JAN: 個数}
   b. 普通单品 → 拆出 (sku_jan, sku_qty)（没有 '-' 时 sku_qty 按 1 处理），
      若 sku_jan 为 13 位：输出 {sku_jan: sku_qty × 個数}
   c. product_dict 两键联查（兜底，见下节）

3. 以上均失败 → 记录到 err_export.csv，人工处理
```

**后缀 `-0` 规则说明**：`商品番号` 后缀为 `0` 表示该商品在 RMS 上注册为"多种套装/选项任选其一"，
此时 `商品番号` 本身的 JAN **不能直接采用**，必须改用 `システム連携用SKU番号` 解析出的 JAN
（该列由 RMS 管理员针对客户实际选择的那个选项填写）。

---

## 四、`product_dict.json` 兜底字典

当 `システム連携用SKU番号` 解析后 sku_jan 也不是 13 位时（极少数情况），
查此字典：

```json
{
  "商品番号_barcode": {
    "sku_jan（システム連携用SKU番号解析后的barcode部分）": "13位JAN"
  }
}
```

实际需要手动补录的情况：`err_export.csv` 出现新记录时，
查 RMS 获取对应 JAN，添加到此文件。

## 五、各场景汇总

| `商品番号` | `システム連携用SKU番号` | `個数` | 输出 |
| --- | --- | --- | --- |
| `4534551421008` | `4534551421008` | 1 | `{4534551421008: 1}` |
| `4534551421008-2` | （任意/空） | 3 | `{4534551421008: 6}` ※1 |
| `4968027302575-0` | `4968027302742-1` | 4 | `{4968027302742: 4}` ※2 |
| `4971710` | `4971710376227-1` | 2 | `{4971710376227: 2}` |
| `4971710` | `4971710376227-2` | 1 | `{4971710376227: 2}` |
| `4971710` | `4971710376227-7002-2` | 1 | `{4971710376227: 1, 4971710377002: 1}` |
| `4971710` | `4971710376227-7002-2` | 3 | `{4971710376227: 3, 4971710377002: 3}` |
| `decorte-tpp` | `4971710374971-1` | 1 | `{4971710374971: 1}` |

※1 `商品番号` 后缀为 1~3 位非0数字 → 直接作为 set_count，`システム` 列不参与计算。

※2 `商品番号` 后缀为 `0` → 即使 barcode 本身是完整13位JAN，也不能直接采用，
必须改用 `システム連携用SKU番号` 解析结果（该商品在RMS上是"多选项"商品，
`システム連携用SKU番号` 记录的才是客户实际选择的那个JAN）。

## 六、参考实装

核心解析逻辑已抽取为独立纯函数模块 **`jan_resolver.py`**（无 pandas / IO 依赖）：

```python
from jan_resolver import resolve_jan_quantities

results = resolve_jan_quantities(product_no, sys_sku_raw, order_count, product_dict)
# results: [(JAN, 数量), ...]；无法解析返回 []
```

- `product_no`：`商品番号` 原始字符串
- `sys_sku_raw`：`システム連携用SKU番号` 原始字符串（可为 `""`）
- `order_count`：`個数`（int）
- `product_dict`：兜底字典 `{barcode: {sku_jan: JAN}}`（见上节，可传 `{}`）

快递文件生成项目中的 `merge_tool.py` → `merge_product_columns(row, product_dict)`
是基于此函数的薄封装：将结果格式化为 `["JAN-数量", ...]` 字符串列表，
并在 `results` 为空时写入 `err_export.csv`。

**库存/采购模块集成方式**：直接复制 `jan_resolver.py` + `product_dict.json` 到 WMS 项目，
对每一行订单调用 `resolve_jan_quantities(...)`，得到的 `(JAN, 数量)` 列表即可用于：

- 查询该 JAN 是否在库存表中存在 → 不存在则标记给采购模块新增 SKU/JAN
- 存在但库存数量 < 本次所需数量 → 标记为"库存不足，需第三方调货"，并记录缺口数量

## 七、与快递子系统的关系（架构备注）

本项目（快递文件生成）将作为乐天 WMS 的子系统。`jan_resolver.py` 是两个系统的共同基础模块，
两边各自维护一份拷贝；若后续逻辑变更（如本文档新增的 `-0` 规则），
需要同时更新两个项目中的 `jan_resolver.py` 和 `product_dict.json`，避免逻辑漂移。
